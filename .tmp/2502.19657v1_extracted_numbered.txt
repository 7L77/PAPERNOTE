0012: Abstract
0013: Neural Architecture Search (NAS) is a powerful automatic alternative to manual design of a
0014: neural network. In the zero-shot version, a fast ranking function is used to compare architec-
0015: tures without training them. The outputs of the ranking functions often vary significantly
0016: due to different sources of randomness, including the evaluated architecture’s weights’ ini-
0017: tialization or the batch of data used for calculations. A common approach to addressing
0018: the variation is to average a ranking function output over several evaluations. We propose
0019: taking into account the variation in a different manner, by viewing the ranking function
0020: output as a random variable representing a proxy performance metric. During the search
0021: process, we strive to construct a stochastic ordering of the performance metrics to deter-
0022: mine the best architecture. Our experiments show that the proposed stochastic ordering
0023: can effectively boost performance of a search on standard benchmark search spaces.
0024: 1 Introduction
0025: Zero-shot Neural Architecture Search (NAS) is a way to perform selection of a neural architecture without
0026: training every candidate architecture. Recently, zero-shot NAS has been applied to a variety of practical
0027: tasks, including medical image processing (Wang et al., 2024), sequence processing (Serianni & Kalita, 2023),
0028: language model architecture enhancements (Javaheripi et al., 2022) and general image classification (Jiang
0029: et al., 2024; Li et al., 2022).
0030: In general, a zero-shot approach consists of two parts: a ranking function and a search algorithm (Xu et al.,
0031: 2021; Rumiantsev & Coates, 2023; Cavagnero et al., 2023a; Li et al., 2022; Jiang et al., 2024). The ranking
0032: function aims to capture the quality of a particular candidate architecture in the form of a scalar value,
0033: which allows two architectures to be easily compared (Abdelfattah et al., 2021). A search algorithm uses a
0034: ranking function to determine the best architecture over the givenarchitectural search space. For this work,
0035: we view a search space as a combination of a feasible architecture set and a dataset of labelled training,
0036: validation, and test samples.
0037: While the zero-shot approach is the fastest way to find an architecture that performs well, it suffers from
0038: variations in the output of the ranking function. In order to assess an architecture as quickly as possible,
0039: most zero-shot techniques sample a single batch of data, and randomly initialize the architecture. The
0040: ranking function then uses the batch of data and the initialized network to derive a scalar score. The
0041: batch selection and initialization thus induce considerable randomness in the output. There are several
0042: data-agnostic ranking functions, including SynFlow (Abdelfattah et al., 2021) and LogSynFlow (Cavagnero
0043: et al., 2023a), which use a batch ofconstant unit inputs, and Zen-Score (Lin et al., 2021), which uses a batch
0044: of randomly generated data as an input. Thus, there is still randomness, and the same problem remains.
0045: Typically, zero-shot techniques address this issue by averaging the outputs of the ranking function over
0046: multiple batches of data.

0066: • We introduce a ranking function variance analysis framework for zero-shot NAS researchers to analyse
0067: existing and newly designed ranking functions.
0068: • We use the variance of a ranking function to statistically compare architectures, and show that this
0069: enhances the performance of random and evolutionary search algorithms for zero-shot NAS for a variety
0070: of commonly-used ranking functions and architectural spaces.
0071: 2 Background
0072: 2.1 Search Algorithms
0073: There are three common choices of search algorithms that are used in Zero-Shot NAS: random search, prune-
0074: based search and evolutionary search.Random search is the search algorithm with the least computation
0075: overhead. It randomly samples a subset of architectures from the search space, and then selects the one
0076: with the highest rank according to the chosen ranking function. The size of the sampled subset is the main
0077: hyperparameter that controls the trade-off between search speed and efficacy. Due to its simplicity, it is a
0078: common choice (Lopes et al., 2021; Xu et al., 2021; Mellor et al., 2020; 2021; Mok et al., 2022).
0079: Evolutionary search procedures are based on genetic algorithms. They start with randomly selected
0080: architectures, then generate mutations, and then select a subset that are ranked the highest according to
0081: the ranking function output. The procedure is then iterated. Mutations can be based on replacement of
0082: architectural operators or involve direct manipulations of the architecture encoding. Evolutionary search is
0083: slower and more challenging to implement than random search, but the results it produces are more stable.
0084: Variations of this search technique are employed more often in recent works (Lin et al., 2021; Akhauri et al.,
0085: 2022; Li et al., 2022; Mok et al., 2022; Cavagnero et al., 2023a;b; Yang et al., 2023; Wang et al., 2024). Often,
0086: the ranking function is applied to multiple batches of data, with the results averaged, in order to provide
0087: a more stable score. This procedure is usually performed once, when the architecture is first encountered
0088: during the search, with the calculated score cached for later use.
0089: The third type of search algorithm isprune-based search. It iteratively removes edges from the hypergraph
0090: that can be used to represent all architectures. By estimating the difference in ranking function output for
0091: the architectures represented by the hypergraph with and without a given edge, the algorithm can choose
0092: the best edge to prune. This process is repeated until the hypergraph represents a single architecture. This
0093: type of search introduces significant computational overhead as it requires to evaluate the entire hypergraph
0094: for every unpruned edge (Chen et al., 2021). It is the most challenging to implement and, therefore, is rarely
0095: adopted (Chen et al., 2021; Rumiantsev & Coates, 2023; Jiang et al., 2024).

0099: 2.2 Zero-shot ranking functions
0100: Many zero-shot ranking functions have been proposed, and we now review some of the most prominent.
0101: Abdelfattah et al. (2021) repurposed metrics previously employed in neural network pruning. The saliency
0102: metrics Snip (Lee et al., 2019),Grasp (Wang et al., 2020),Fisher (Turner et al., 2020) andSynFlow (Tanaka
0103: et al., 2020) measure the impact of removing a specific parameter or operation from the network. In adapting
0104: these to construct a zero-shot ranking function, Abdelfattah et al. (2021) sum over all of the architectural
0105: parameters, and thus construct a measure of the fitness of the entire architecture. Cavagnero et al. (2023a)
0106: introduced a logarithmic version ofSynFlow named LogSynFlow. Applying a logarithm suppresses some of
0107: the gradient explosions that can unduly influence the score.
0108: The first two ranking functions designed specifically for the zero-shot setting were the Eigenvalue
0109: score (Mellor et al., 2020) and the Hamming distance between the ReLU regions (Mellor et al., 2021).
0110: Based on the Neural Tangent Kernel (NTK) (Jacot et al., 2018), theEigenvalue score uses the eigen-
0111: values of the gradient correlation matrix to predict the learning capabilities of an architecture. TheReLU
0112: Hamming distance is constructed as a determinant of the kernel of Hamming distances betweent activation
0113: patterns of ReLU, where a pattern is represented as a string of zeros and ones. An alternative measure is
0114: the number of activated ReLU regions for a sampled batch of data (Xiong et al., 2020). The underlying
0115: conjecture is that activation of more regions allows the network to learn more complex data dependencies.
0116: Similarly, Peng et al. (2024) proposes to count the number of unique ReLU activation patterns across the
0117: data dimension. The Entropic Score (Cavagnero et al., 2023b) is also based on ReLU activations.
0118: Several other ranking functions are based on the NTK, including (i) the mean value of the NTK (Xu et al.,
0119: 2021); (ii) the Frobenius norm of the NTK (Xu et al., 2021); and (iii) the negative condition number of the
0120: NTK (Chen et al., 2021). The NTK can introduce undesirable computational overhead, so other researchers
0121: have explored alternative metrics. Rumiantsev & Coates (2023) show that the Frobenius norm of the Neural
0122: Network Gaussian Process (NNGP) kernel can act as an effective ranking function. Jiang et al. (2024) focus
0123: on the Pearson correlation kernels of the internal representations of an architecture, and compute the sum
0124: of the smallest eigenvalues of these kernels. For all of these proposed ranking functions, the required kernels
0125: are estimated using a random batch of data.
0126: Since most benchmark search spaces focus on the classification task, some ranking functions incorporate the
0127: available label information. Label-Gradient Alignment (LGA) (Mok et al., 2022) constructs a metric based on
0128: both the class similarity matrix and the NTK. Lopes et al. (2021) proposeEPE-NAS, which uses the gradient
0129: correlation matrix, and is thus similar to theEigenvalue score , but is computed class-wise. For practical
0130: reasons, both LGA and EPE-NAS are computed using a random batch of data.
0131: Some ranking functions are based on the gradients of the trainable weights.SweetGrad counts the number of
0132: gradient norms in a predefined interval (Yang et al., 2023). Li et al. (2022) sum the coefficients of variations
0133: associated with the weight gradients to computeZiCo.
0134: 3 Problem statement
0135: A neural architecture search space can be represented as a set of feasible architecutres{archi}[i=1..NSS ] of
0136: size NSS. Associated with the architecture space is a datasetD consisting of pairs of featuresx∈Xand
0137: labels y∈Y. In this work we consider computer vision tasks where an architecturearchi is associated
0138: with a final accuracy after training, labelledacci. A zero-shot ranking function assigns a scalar score to
0139: each architecture: si = r(archi,D). A search is then conducted using these scalar scores, with the goal of
0140: selecting the most accurate architecture.
0141: As discussed above, almost every zero-shot ranking function has an element that is evaluated over a batch
0142: of input data (e.g., the NTK, gradient coefficients of variations, or logit similarity matrix). To improve the
0143: stability of the output, most ranking functions are averaged over multiple random batches. Our goal is to
0144: use the multiple ranking function outputs in a more effective way.

0148: In subsequent sections, we explore how sensitive different ranking functions are to the random selection of
0149: the input data used in their calculation. We point out how commonly used statistical methods allow us to
0150: overcome such sensitivity in order to improve the performance of a zero-shot approach.
0151: 4 Methodology
0152: This section describes our study into the variability of ranking functions. First, we introduce a variation
0153: metric for assessing the variability of a ranking function for a particular search space and batch size. Sec-
0154: ond, we present our zero-shot search setup and explain how the variation metric can be used to improve
0155: performance.
0156: 4.1 Ranking function variation
0157: It is common for ranking functions to use a real training dataset. In order to assign a score to an architecture,
0158: a batch of data is randomly drawn from the dataset. The ranking function scores vary from batch to batch,
0159: with the extent of the variation being potentially dependent on the batch size, the dataset, and the task.
0160: We measure ranking function variation over the entire search space by computing, for each architecture, an
0161: empirical coefficient of variation for the ranking function output over randomly sampled batches of input
0162: data. We then average over the architectures. To ensure the coefficient of variation is an appropriate metric,
0163: we have specified a “meaningful zero” for the ranking functions. We discuss this in detail in Appendix G
0164: Denoting the batch size byB and the number of sampled batches byV, and recalling that there areNSS
0165: architectures in the search space, we define the ranking function variation as:

0176: Typically, if we know that a ranking function is highly variable with respect to the input data, then this
0177: motivates us to enhance the robustness by averaging over multiple random batches. On the other hand, if
0178: a ranking function exhibits less variability, then a score can be taken from a reduced number of randomly
0179: drawn batches. This leads to a faster search, because ranking function evaluation tends to be the most
0180: computationally expensive element of zero-shot NAS.
0181: 4.2 Search algorithm
0182: Zero-shot architecture search uses a ranking function as a proxy to evaluate the quality of an architecture.
0183: In order to compare two architectures, a selected ranking function is evaluated on both of them and the
0184: outputs of the ranking function are compared. Assuming positive correlation between accuracy and the
0185: adopted metric, the architecture with the higher corresponding metric is considered to be better. The
0186: ranking function outputs are subject to randomness, due, for example, to random architecture initialization
0187: or random selected input batch. A common technique to mitigate randomness is averaging: a ranking
0188: function is evaluated multiple times on the same architecture and the output is averaged over the evaluations.
0189: Instead of averaging, the key component of our approach is a statistical comparison of the architectures. We
0190: treat multiple evaluations of the ranking metric as multiple observations of a random variable. We then test
0191: for stochastic dominance. Viewing the ranking function metrics for two compared architectures as random
0192: variables,X andY, we test whetherp(X >k)≥p(Y >k ) for allk (andp(X >k)>p (Y >k ) for somek).
0193: This test differs from comparing the mean performance,¯X >¯Y. Stochastic dominance implies ¯X >¯Y, but
0194: the converse is not true. Our testing process thus targets a stricter performance discrepancy. This makes
0195: the search more effective and efficient, because we navigate towards architectures that are more likely to be
0196: genuinely superior.

0200: To perform the comparison, we use the U-value from the Mann-Whitney U-test (Mann & Whitney,
0201: 1947). This statistical test assesses the null hypothesis that both sets of observations are drawn from
0202: the same distribution. The alternative hypothesis (for a one-sided test) is that the random variable
0203: X is stochastically greater than the random variable Y. By comparing the Mann-Whitney U values for
0204: the two samples, we can identify one as a superior architecture. We declare such superiority only if
0205: the null hypothesis can be rejected at a predetermined significance level (e.g., 0.05). If this is not the
0206: case, we cannot meaningfully differentiate between the architectures in terms of predicted performance.

0475: approach over this naive scheme is indeed considerably greater for these ranking functions. Although a naive
0476: incorporation of the coefficient of variation can improve results of the search, its impact is inconsistent;

0589: 5.3 Evolutionary search
0590: We also compare the performance of the standard averaging approach with that of the proposed statistical
0591: evaluation when each is used in conjunction with evolutionary search. As before, the 5% significance level is
0592: used for rejecting the null hypothesis of the Mann-Whitney U-test. We apply caching for both approaches.
0593: In the case of averaging, we cache the average of the ranking function output over 10 evaluations. For the
0594: statistical evaluation we cache the individual outcomes of the 10 evaluations. Whenever two architectures
0595: are statistically compared, we store the outcome, so that the procedure does not need to be repeated.
0596: Table 2 presents the evolutionary search results on the NAS-Bench and TransNAS-Bench search spaces.
0597: We demonstrate results on Eigenvalue score and ReLU Hamming distance. We also include an ensemble
0598: of those that is formulated as a sum with components individually MinMax normalized over the observed
0599: architecture ranks. We observe that statistical evaluation consistently improves the search quality. Although
0600: the particular improvement depends on the combination of the search space and ranking function, we did
0601: not observe a case where applying a statistical evaluation leads to a poorer search result. We do not
0602: observe a significant difference between the improvement for low- versus high-variance ranking functions. In
0603: our experiments with evolutionary search, Eigenvalue score (low variance) has the average improvement of
0604: 0.73, whereas for RELU dist (high variance) the average improvement is 0.60. We observed that FreeREA
0605: demonstrates the best results in most cases, thus emphasising the importance of crossover in evolutionary
0606: search. However, in the averaging case the difference between FreeREA and REA is typically less than
0607: one standard deviation. This may make REA more suitable for some practical applications as it requires
0608: only to define a mutation operation. FreeREA experiences the largest performance increase with statistical

0628: 6 Conclusion and Future work
0629: In this work, we proposed a statistical approach for comparing architectures during a NAS zero-shot search
0630: procedure. Through experiments using both random and evolutionary search, we provided evidence that
0631: the proposed approach leads to better search results than simply averaging the outcomes of a batch of
0632: evaluations. We also investigated the variation of a wide selection of zero-shot ranking functions for several
0633: search spaces and observed that those which exhibit less variability tend to produce better and more stable
0634: search outcomes. Given these observations, we can recommend that zero-shot search is conducted using
0635: statistical architecture comparison.
0636: Every ranking function evaluation for a given architecture can be considered equivalent to sampling from the
0637: distribution. In our work, we handled such sampling with statistical testing in order to retain compatibility
0638: with the existing pipelines and search agorithms. Formulating a zero-shot neural architecture search as a
0639: stochastic optimisation problem that interprets the output of a ranking function not as a score, but as a
0640: distribution could be a potential research direction.
0641: We have observed that the statistical method does not consistently benefitial for data-agnostic SynFlow and
0642: LogSynFlow, while their performance is competitive with other ranking functions discussed in the paper.
0643: This suggests that an alternative strategy to the statistical approach is to develop techniques that are
0644: minimally affected by the data batch and initialisation.
0645: An additional important source of performance variability that we have not discussed is the search space.
0646: Our experiments show that the suitability of a ranking function is highly dependent on the search space.
0647: For one search space, a particular ranking function might be the best choice; on a different search space,
0648: it might perform poorly. Due to this variability, we cannot predict how our conclusions will generalise to a
0649: new search space.

0762: NAS-Bench-201(Dong&Yang,2019)has15625architecturesinthesearchspaceandprovidesperformance
0763: for CIFAR-10, CIFAR-100, and ImageNet-16-120. It has 5 cells. Every cell has 4 nodes. The list of
0764: operations includes “zeroize”, which is equivalent to cutting the connection, “skip connection”, “3x3 conv”,
0765: “1x1 conv” and “3x3 avg-pooling”. This is the most popular benchmark as it is relatively small and contains
0766: measurements for several datasets.NAS-Bench-360(Tu et al., 2022) extends this benchmark by additional
0767: datasets. In this work, we are using the NinaPro DB5 dataset from NAS-Bench-360. For consistency, we
0768: refer to it as NAS-Bench-201 (NinaPro).
0769: T ransNAS-Bench-101(Duan et al., 2021) is a tabular benchmark that consists of two search spaces
0770: evaluated on seven tasks. The architecture of the neural network in the benchmark is a composition of a
0771: searchable encoder and a decoder tailored to specific tasks. The macro search space suggests searching for
0772: the number of blocks, the size for downsampling, and the count of output channels per block. The micro
0773: search space is composed of four nodes, similarly to NAS-Bench-201. Four operations are used: “zeroize”,
0774: “skip connection”, “3x3 conv block” and “1x1 conv block”, where every convolution block is a sequence of
0775: ReLU, convolution, and batch normalisation. In our project, we are only using the micro search space that

1043: We observe that caching, which is typically applied primarily with the motivation of reducing computational
1044: overhead, always improves the search performance. Moreover, caching makes the results more stable in the
1045: majority of the cases, i.e., reduces the standard deviation. We conjecture that this performance improvement
1046: arises because the use of caching means that previous search decisions are not overturned. In the case
1047: of ‘on-the-fly’ evaluation, the search can initially decide that an architecture is inferior, but then reverse
1048: that decision when the architecture is encountered a second time. That can be clearly seen for a greedy
1049: evolutionary search, which is designed to maximise the number of re-evaluations. The ‘hybrid’ approach
1050: improves the stability, but results in poorer performance compared to the ‘cached’ approach. The efficiency
1051: of the search is improved by committing to a decision. The observed performance improvement is less
1052: for low-variance ranking functions, as their output is more consistent, leading to fewer changed decisions.

1061: We have assessed the concept of a meaningful zero for each ranking function listed above. Here are our
1062: findings.

1066: • The Frobenius norm of NTK is always bigger than zero, and since it is a norm, it clearly has a meaningful
1067: zero.
1068: • The mean value of NTK is a mean of all elements for a positive semidefinite matrix. Since the NTK is
1069: positive-semidefinite (Jacot et al., 2018), the mean value is always non-negative, implying that there is a
1070: meaningful zero.
1071: • Label-Gradient Alignment is computed from the NTK H ∈RB×B and label-alignment matrix L ∈
1072: {0, 1}B×B (see (Mok et al., 2022) for details). Since the NTK is positive-semidefinite, LGA is bounded
1073: from below. The metric has a meaningful zero.
1074: • The Eigenvalue score is computed using the eigenvalues of NTKH, which are all non-negative. The
1075: Eigenvalue score has a meaningful zero (corresponding to the extreme case of a zero-valued kernel).
1076: • The ReLU Hamming distance is based on the comparison of ReLU activation patterns for different inputs.
1077: Since it is evaluated based on distances, it has a meaningful zero corresponding to the case when the
1078: activations match exactly for all inputs.
1079: • EPE-NAS score is the class-wise sum of absolute values, which is non-negative. It has a meaningful zero.
1080: • The condition number of NTK is always positive and greater than or equal to one. Although the zero
1081: value cannot be achieved, it is a meaningful value in the sense that the zero point is not arbitrary. We
1082: can shift or scale the condition number and retain its fundamental meaning.
1083: • Fisher is a layer-wise sum over squares. It is non-negative and has a meaningful zero.
1084: • Snip is a layer-wise sum over absolute values that is non-negative and has a meaningful zero.
1085: • The implementations of both SynFlow and LogSynFlow use the layer-wise summation of absolute values.
1086: These values are non-negative, and they both have a meaningful zero.
1087: • MeCo is a layerwise sum of the smallest eigenvalues of Pearson correlation matrices for feature maps; as
1088: the matrices are positive-semidefinite by design (see (Jiang et al., 2024) for details), MeCo is non-negative
1089: and has a meaningful zero.
